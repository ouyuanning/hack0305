🎯 架构说明

  • 左侧闭环：数据源 → AI采集 → MO仓库 → AI提炼 → 知识库 → 自动提Issue → 交互层 → 创建Issue → 回到GitHub（形成完整闭环）
  • 右侧分支：从MO仓库和知识库分出两个功能：① 历史清洗（存试验库）② 项目看板（输出到企微和邮箱）
  • 核心连接：MO仓库是中心，向右分支出项目库和历史清洗；知识库向下支撑自动提Issue，也向右支撑历史清洗

分支功能说明

  • 分支1 - 历史清洗：从MO仓库读取历史Issue，基于知识库的规则进行清洗，打上标签后存入试验库（另一个独立库）
  • 分支2 - 项目看板：从MO仓库拉取特定Tag的条目，存入项目库（单独的库），通过AI分析和统计模块，生成看板后输出到企微通知和邮箱

数据库说明

  • MO主库：核心数据中台，存储所有Issue快照，是整个系统的数据基础
  • 项目库：从主库中拉取特定Tag的条目（如project/问数深化），用于项目看板和每日分析
  • 试验库：历史Issue经过清洗和打标签后的数据，用于试验新规则和新功能

💡 关键技术点

🔗 数据源同级

GitHub仓库和代码仓库是同一层级的数据源，它们之间有关联（待实现），最终都会通过AI处理进入MO仓库

🤖 两次AI处理

第一次：数据采集时的AI处理（清洗/提取）
第二次：生成知识库的AI处理（知识提炼）
贯穿全程：核心功能中的AI处理（判断/分析/生成）

🔍 匹配方式待优化

自动提Issue时，根据要点与MO仓库中的Issue匹配，注意：不一定是看向量，搜索模式待优化（可能是关键词、标签、规则等多种方式）

📱 交互层差异

企微：只能获取文字+图片
Cursor：可以额外获取浏览器/窗口信息（代码文件、报错、Git分支等）
